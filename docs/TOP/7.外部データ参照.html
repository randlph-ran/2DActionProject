外部データ参照 ＋ セーブ/ロード

Enemy / Player / Item のCSVデータ参照、ストーリーテキストの外部参照、セーブ/ロード対応の
「やること」と「対応順番」のたたき台。

全体の順番（推奨）

Phase A: 基盤づくり
   └ CSV読み込みの共通仕組み（フォーマット決定 ＋ 共通ローダー）

Phase B: 各データのCSV化（影響が小さい順）
   1. ItemData    ← 独立度が高く一番安全。仕組みの検証台に最適
   2. EnemyData   ← 種類ごとに独立。項目数は多いが影響は閉じている
   3. PlayerData  ← Controller/Combat/Healthに跨る。最後に回す

Phase C: ストーリーテキストの外部参照（独立。B と並行でも可）

Phase D: セーブ/ロード（B の Player/Item 構造が固まってから）

各Phaseの中身

Phase A：基盤

  • CSVの置き場所を決める（Resources/ か StreamingAssets/ か）
  • 共通のCSVパーサ/ローダーを1つ作る（各データで使い回す）
  • 「CSV → ScriptableObject / 構造体」への流し込み方針を決める

Phase B：CSV化（1 → 2 → 3 の順）

  • 各Dataの現状 SerializeField のうち、外部化する項目を洗い出す
  • ItemDataで仕組みを確立 → Enemy → Player へ展開
  • Playerは分割済みの各コンポーネント（Controller / Combat / ItemAction / Health）参照に注意して最後に

Phase C：ストーリーテキスト外部参照

  • Inspector直書き → テキスト/CSVからの読み込みに差し替え
  • ※将来のイベントタグ・タイプライター表示の前段になる（今回はデータ化のみ）

Phase D：セーブ/ロード

  • セーブ対象を確定（HP・所持品・進行度など）
  • 保存形式（JSON等）と保存先を決定
  • GameData の既存構造を土台に拡張

最初に決めるべきこと（次回冒頭で相談）

  1. データ形式：CSV 前提でよいか（JSON / ScriptableObject 併用の可否）
  2. CSVの置き場所：Resources / StreamingAssets（実行後に差し替えたいかで変わる）
  3. セーブ形式：PlayerPrefs か ファイル(JSON) か

この順番の理由

  • 後戻りの大きいPlayerを最後に、検証しやすいItemを最初に
  • セーブ/ロードはデータ構造が固まってから（先にやると二度手間）
  • ストーリーテキストは独立しているので、詰まったら差し込める緩衝材

  • ノックバックのState化（PlayerState.Knockback）完了
